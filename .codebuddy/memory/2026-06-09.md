# 2026-06-09 工作日志

## RustFS 缓存兜底修复
问题：API 挂了后 RustFS 无法可靠兜底播放已缓存歌曲
根因：
1. `DownloadService` 缓存命中时存入 DB 的是 7 天有效预签名 URL，过期后失效
2. `getPlayUrl`/`getPlayInfo` 不优先检查 RustFS，每次都先调 API
3. `SongController.stream` 仅通过 HTTP 代理远程 URL，没有 RustFS 直读能力
4. 浏览器下载文件名只有 sourceId 数字，无法辨识

修复内容：
- `StorageService`：新增 `getDirectUrl()` 返回不过期的直接访问 URL
- `SongService.getPlayUrl/getPlayInfo`：优先级改为 RustFS缓存 → API → DB兜底
- `DownloadService`：缓存命中时用 `getDirectUrl()` 替代 `getPresignedUrl()`
- `SongController.stream`：新增 RustFS 直读路径（`StorageService.getObject`），远程代理为降级方案
- `DownloadController.fileDownload`：下载文件名改为 "歌手 - 歌曲名.mp3"
- `SongService`：新增 `getBySourceId()` 方法

## 单曲循环播完暂停修复
问题：单曲循环模式播完一首歌就暂停了
根因：`onEnded` 中用 `audio.currentTime=0; audio.play()` 重播，但浏览器 autoplay 策略阻止了无用户交互的 play() 调用
修复：改用浏览器原生 `audio.loop = true`，单曲循环时不触发 ended 事件，浏览器自动从头播放，不受 autoplay 限制
改动：PlayerBar.vue - syncPlayMode/playCurrent/setAudioSrc 中同步 audio.loop，onEnded 移除 single 分支

## 移动端播放状态不同步重构
问题：移动端歌曲播放资源与底部栏/歌词页信息不匹配，刷新后最近播放和收藏加载慢
根因：
1. `PlayerBar` 仅桌面端渲染 → 移动端缺 `vibeAudioSetSrc`/`onEnded`/`audio.loop`
2. `MBottomPlayer` 的 `_onSongChange` 每次删除+追加歌曲 → 队列索引错位
3. 6+ 组件各自独立管理 `audio.src` 和 localStorage → 状态分散易失同步

修复方案：创建 Pinia player store 统一管理
新增文件：
- `src/stores/player.js` - 集中管理 audio/queue/播放模式/切歌/进度/音量

重构文件：

- `PlayerBar.vue` - 使用 store 替代内部状态
- `MBottomPlayer.vue` - 使用 store，消除独立队列管理
- `MPlayerView.vue` - 使用 store，消除独立 audio.src 设置
- `MHomeView.vue` - 使用 `player.playSongFromApi()`
- `MSearchView.vue` - 使用 `player.playSongFromApi()` 和 `player.addToQueue()`
- `MRecentView.vue` - 使用 `player.playSongFromApi()`
- `MLikesView.vue` - 使用 `player.playSongFromApi()`
- `MQueuePopup.vue` - 使用 store 的 queue/currentIdx/playIndex/removeFromQueue

核心改进：
- 单一数据源：所有播放状态集中在 Pinia store
- 移动端/桌面端共享同一套播放逻辑
- `song-change` 事件由 store 统一派发
- `audio.loop` 由 store 统一管理
