### 7. I/O 优化全部实施
基于 IO_ANALYSIS.md 分析结果，实施了所有 P0/P1/P2 修复：

**P0 修复：**
- `DownloadService`: 事务拆分 → HTTP/文件 I/O 在事务外完成（30s+），DB 持久化在短事务内（<50ms）
- `RecommendService`: `markOfflineStatus()` 从 N 次 MinIO statObject → 1 次 DB 批量查询 song.url

**P1 修复：**
- `AssistantController`: 新增 POST /stream SSE 端点，逐 token 推送，消除 servlet 线程长时间阻塞
- `SongPlayService`: `isCachedInRustFS()` 增加 Redis 缓存（存在 10min TTL，不存在 30s），减少 MinIO HTTP 调用

**P2 修复：**
- `application.yml`: Redis lettuce pool max-active 8→20, min-idle 0→2
- `StreamUtils`: buffer 8KB→64KB
- `NeteaseApiService`: 删除已废弃的 `downloadSong(byte[])` 方法

修改文件: DownloadService.java, RecommendService.java, AssistantController.java, SongPlayService.java, NeteaseApiService.java, StreamUtils.java, application.yml

### 8. API 调用缓存优化
基于全量 API 调用审计，新增了 4 处 Redis 缓存，减少对 musicapi 的无谓调用：
- **歌词缓存** (`SongController.lyric`): TTL 365天，歌词永不变，一次拉取永久复用
- **首页轮播缓存** (`SongController.banner`): TTL 2h，首页频繁刷新不再重复请求
- **歌单详情缓存** (`PlaylistController.detail`): 按 source:id 键，TTL 6h
- **推荐歌单缓存** (`PlaylistController.recommend`): 缓存原始 30 个精选，每次随机取 6
- **死代码清理** (`NeteaseApiService`): 删除 4 个无调用者的方法 (search/aggregatedSearch/getSongDetail/旧downloadSong)

修改文件: SongController.java, PlaylistController.java, NeteaseApiService.java
