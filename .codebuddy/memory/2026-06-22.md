# 2026-06-22 MySQL 性能优化

## 完成事项
对 vibeMusic 项目进行了全面的 MySQL 性能审查和优化：

### 1. 慢查询日志（新增）
- 修改 `docker-compose.yml`，MySQL 容器 command 中开启了：
  - `slow_query_log=ON`
  - `long_query_time=1`（超过 1 秒即记录）
  - `log_queries_not_using_indexes=ON`
  - `log_slow_admin_statements=ON`

### 2. 索引优化（新增 Flyway V2 迁移）
- 删除 3 个冗余索引：
  - `playlist_song.idx_playlist_id`（被 `uk_pl_song` 前缀覆盖）
  - `user_favorite.idx_user_id`（被 `uk_user_song` 前缀覆盖）
  - `play_history.idx_user_id`（被 `idx_user_played` 前缀覆盖）
- 新增 2 个缺失索引：
  - `song.idx_name`（歌曲名前缀索引，随机推荐/本地搜索使用）
  - `song.idx_created_at`（排序查询使用）

### 3. SQL 优化
- 移除 `PlaylistMapper.java` 中与 `PlaylistMapper.xml` 重复的 @Select SQL
- `PlaylistMapper.xml` 添加了详细的索引依赖注释说明

### 审查发现
- 项目使用 MyBatis-Plus + HikariCP，未引入 Druid 监控
- 搜索走 Redis → ES → API 三级缓存，绕过了 MySQL 查询
- `SongMapper.findRandomSongs` 用 OFFSET 方式取随机数据，大表时需关注
- `PlaylistMapper.listPlaylistsWithStats` 的关联子查询配合 `idx_pl_added` 索引，在单用户 ≤20 个歌单的场景下高效
