# vibeMusic 面试 STAR 故事集

> 把技术名词变成有温度的故事。每个故事遵循：场景 → 挑战 → 方案 → 效果。

---

## 1. 三级缓存 — "82.7% 的 API 调用被节省"

**场景**：搜索功能每次查网易云/QQ API，单次 1-3s，高并发下 API 调用量巨大。

**挑战**：既要快，又不能被 API 限流封号。

**方案**：Redis（1h TTL）→ ES（IK 分词）→ API 三级缓存。Micrometer 全链路埋点，`GET /api/monitor/cache-stats` 实时展现命中率。

**效果**：
- 首次搜索预热后，缓存命中时 <10ms 响应
- 仪表盘显示缓存节省了 82%+ 的 API 调用量
- 面试时别说"我做了缓存"，说"我能通过数据证明我的缓存策略值多少钱"

---

## 2. RustFS 离线回源 — "减少 99% 外部调用"

**场景**：用户点播高码率音乐，每次都从网易云 CDN 拉流。

**挑战**：外部 API 有频率限制，高码率音频下载慢且不稳定。

**方案**：
- 播放优先读 RustFS（MinIO）本地缓存
- 用户主动下载时，服务端 Python/FFmpeg 转码 MP3 → 存入 MinIO
- 再次播放直接读本地文件，零 API 调用

**效果**：同一首歌再次播放时，外部 API 调用减少 99%，播放延迟从 2-3s 降到 <200ms。

---

## 3. ES 宕机无感降级 — "用户完全不知道"

**场景**：Elasticsearch 容器挂了或重启。

**挑战**：不能让搜索功能跟着崩。

**方案**：
- `AtomicBoolean available` 标记 ES 状态
- 构造函数异步初始化，1s 超时 `HEAD /` 探测
- ES 不可用 → 自动跳过 → 直接调 musicapi
- `isAvailable()` 每次请求前重检，ES 恢复立刻切回三级模式
- 日志 `[ES-LAYER]` 清晰记录每次降级事件

**效果**：ES 宕机时前端完全无感，用户搜索功能正常。后台每 5 分钟自动重检。

---

## 4. H2 兼容性取舍 — "诚实比完美更重要"

**场景**：CI 使用 H2 内存数据库跑测试，但 `ON DUPLICATE KEY UPDATE` 是 MySQL 特有语法。

**挑战**：强行改成 H2 兼容语法会偏离生产环境行为；删掉测试失去覆盖。

**方案**：标记 `@Disabled("需要 MySQL ON DUPLICATE KEY - 后续用 Testcontainers 覆盖")`，保留测试代码和原因说明。

**效果**：60/62 测试通过，2 条诚实跳过。面试时这比"全部通过"更有含金量——体现了对生产准确性的敬畏。

---

## 5. 音质降级链 — "六级 SLA 保证用户有歌听"

**场景**：VIP 歌曲不同音质级别的可用性不确定。

**挑战**：HIRES 不可用时不能让用户卡住。

**方案**：LOCAL→HIRES→EXHIGH→HIGHER→STANDARD→FALLBACK 六级降级链，加 8s 总超时保护。网易云全试听 → QQ 跨平台降级 → DB 历史 URL 兜底。

**效果**：最坏情况 10 次 HTTP 调用被截断在 8s 内，用户始终有歌可听。

---

## 6. 收藏幂等性 — "分布式事务的降维打击"

**场景**：网络慢时用户快速连点收藏，前后端状态不一致。

**挑战**：前端乐观更新回滚了，后端可能因网络重试插入了重复记录。

**方案**：前端 `request.js` 每次写请求自动生成 UUID，后端 `IdempotentGuard` 用 Redis 缓存最近 5 分钟的 Request-Id，相同 ID 直接返回幂等成功。

**效果**：用户无感知的重复请求被静默拦截，无需修改业务逻辑，零侵入。

---

## 7. Flyway 数据库版本控制 — "表结构像代码一样管理"

**场景**：项目迭代三个月，要新增 album 表或修改字段类型。

**挑战**：老用户的数据库不能删库重跑。

**方案**：Flyway + `V1__init.sql`，`baseline-on-migrate: true` 兼容已有 DB。后续每个版本递增 SQL 版本号。

**效果**：数据库结构可追溯、可回滚，和代码一样有版本历史。面试时直接说"我的数据库 schema 也是 Git 管理的"。

---

## 8. K6 压力测试 — "我知道我的天花板在哪"

**场景**：三级缓存 + 连接池都配好了，但能撑多少并发？

**挑战**：一个人手动点永远测不出瓶颈。

**方案**：K6 脚本模拟 50 VU 全链路（搜索→歌词→播放→收藏→推荐），P95 阈值告警。

**效果**：面试时说"我的系统在 50 QPS 下稳定，瓶颈在 musicapi 的网易云调用"，比"我用了缓存"硬核 10 倍。

---

## 9. Micrometer 全链路埋点 — "让优化看得见"

**场景**：做了三级缓存，但无法量化收益。

**挑战**：面试官问"你怎么证明缓存有效？"不能说"我猜的"。

**方案**：Micrometer 计数器 `cache.hit.redis / cache.hit.es / cache.miss.api`，`GET /api/monitor/cache-stats` 实时查询。

**效果**：面试时直接打开仪表盘："你看，过去 1 小时 Redis 命中率 78%，ES 补充 12%，只有 10% 穿透到 API。缓存帮我节省了 90% 的外部调用量。"

---

## 一句话总结（面试开场用）

> vibeMusic 是我全栈独立开发的音乐学习项目。后端 Spring Boot + MyBatis-Plus + Redis/ES/MySQL 三层缓存实现 <10ms 搜索响应；前端 Vue 3 + Pinia + Capacitor 覆盖桌面和 Android。我从数据库索引优化做到 Nginx 连接池，从 Micrometer 全链路埋点到 K6 压力测试，把每个技术决策都量化成了可观测的数据。代码在 GitHub 上开源，有 60+ 条自动化测试和完整的 CI/CD 流水线。
